# Root Cause Analysis (RCA)

## Incident Information

| Item | Details |
|------|---------|
| Incident ID | INC-001 |
| Severity | P1 |
| Service | Flask Reliability Test Service |
| Date | July 2026 |
| Monitoring Platform | Datadog |
| Status | Resolved |

---

# Executive Summary

A simulated reliability incident was intentionally introduced into the Flask Reliability Test Service using the application's runtime chaos injection feature. The incident caused a sustained increase in HTTP 500 responses, resulting in rapid Error Budget consumption, degradation of the Availability SLO, and activation of multiple Datadog monitors.

The observability pipeline successfully detected, visualized, and alerted on the incident, validating the end-to-end monitoring architecture.

---

# Problem Statement

The application began returning HTTP 500 responses for incoming requests to the `/service` endpoint.

As the failure rate increased:

- Successful request count decreased.
- Failed request count increased.
- Availability SLO degraded.
- Error Budget was consumed rapidly.
- Burn Rate exceeded configured thresholds.
- Multiple monitors entered the Alert state.

---

# Root Cause

The incident was intentionally generated using the application's `/chaos` endpoint.

Chaos injection modified the runtime configuration by increasing the application error rate.

Example configuration:

```json
{
    "error_rate": 100,
    "latency_ms": 0
}
```

This configuration forced requests handled by the `/service` endpoint to return HTTP 500 responses, simulating a complete application outage.

This was an expected behaviour used to validate the monitoring solution.

---

# Detection

The incident was automatically detected by Datadog through custom DogStatsD metrics generated by the Traffic Generator.

Detection occurred in the following order:

1. Custom metrics reflected increased failures.
2. Datadog Agent forwarded metrics.
3. Availability SLO decreased.
4. Error Budget consumption increased.
5. Burn Rate increased.
6. Threshold Monitor entered Alert.
7. Burn Rate monitors entered Alert.
8. Reliability Dashboard displayed degraded service health.

---

# Customer Impact

During the simulation:

- Requests to the `/service` endpoint failed.
- Availability decreased.
- Error Budget consumption accelerated.
- Reliability targets were temporarily violated.

Because the incident was conducted in a controlled test environment, no production users were affected.

---

# Resolution

The incident was resolved by disabling chaos injection.

Runtime configuration was restored:

```json
{
    "error_rate": 0,
    "latency_ms": 0
}
```

Following recovery:

- Successful requests resumed.
- Error Budget consumption stabilized.
- Burn Rate returned to acceptable levels.
- Availability SLO began recovering.
- Monitors returned to the OK state.

---

# Validation

Recovery was confirmed using the Reliability Overview Dashboard.

The following indicators returned to normal:

- Availability SLO
- Burn Rate
- Error Budget
- Success Rate
- Monitor Status

---

# Lessons Learned

The simulation successfully validated:

- Custom metric collection
- DogStatsD integration
- Datadog Agent ingestion
- Availability SLO monitoring
- Error Budget tracking
- Burn Rate alerting
- Dashboard visualization
- Incident response workflow

The monitoring pipeline correctly detected the simulated outage and provided sufficient operational visibility for investigation and recovery.

---

# Recommendations

Future improvements include:

- Expand endpoint-level monitoring.
- Integrate distributed tracing.
- Add log correlation to dashboards.
- Automate incident creation through Datadog integrations.
- Enhance runbooks with service-specific recovery procedures.

---

# Conclusion

This incident demonstrated the effectiveness of the implemented observability solution.

The combination of custom metrics, SLO monitoring, Error Budget tracking, Burn Rate alerting, and centralized dashboards enabled rapid detection, diagnosis, and verification of service recovery, validating the overall reliability monitoring architecture.
