# Purpose

This runbook defines the operational procedure for responding to a **Priority 1 (P1)** incident affecting the Flask Reliability Test Service.

A P1 incident represents a critical reliability event that significantly impacts service availability or rapidly consumes the Availability SLO Error Budget. Immediate investigation and mitigation are required.

---

# Trigger Conditions

This runbook should be followed when any of the following conditions occur:

- Fast Burn Rate Monitor enters **Alert**
- Availability SLO falls below the target
- Error Budget is being consumed at a critical rate
- Continuous HTTP 5xx responses are observed
- Complete or near-complete service outage

# Response Procedure

## Step 1 – Acknowledge the Alert

Immediately acknowledge the alert generated by Datadog.

Record:

- Alert Time
- Monitor Name
- Severity
- Alert Message

---

## Step 2 – Open the Reliability Dashboard

Open the **Reliability Overview Dashboard**.

Review the following widgets:

- Availability SLO
- Availability Error Budget
- Burn Rate
- Monitor Status

Objective:

Determine whether the incident is affecting overall service reliability.

---

## Step 3 – Evaluate Service Health

Review:

- Availability SLO
- Latency SLO
- Burn Rate
- Error Budget Remaining

Questions:

- Is the service still available?
- Is the Error Budget rapidly decreasing?
- Is Burn Rate increasing?

---

## Step 4 – Analyze Traffic

Review:

- Total Requests
- Failed Requests
- Success vs Failure Trend
- Endpoint Traffic

Objective:

Determine whether failures affect all traffic or only specific endpoints.

---

## Step 5 – Identify the Affected Endpoint

Review:

- Top Error Endpoints
- Endpoint Traffic Distribution
- Endpoint Latency Comparison

Determine:

- Which endpoint is producing the highest number of failures?
- Whether the issue is isolated or system-wide.

---

## Step 6 – Investigate the Root Cause

Review:

- Flask application logs
- Datadog Events
- Recent configuration changes
- Recent deployments
- Infrastructure status

Possible causes include:

- Application exception
- Dependency failure
- Network issue
- Configuration error
- Intentional chaos injection (testing)

---

## Step 7 – Mitigate the Incident

Take appropriate corrective action.

Examples:

- Disable chaos injection.
- Restart the application.
- Roll back recent deployment.
- Restore dependent services.
- Correct configuration errors.

---

## Step 8 – Verify Recovery

Confirm that:

- Availability SLO is recovering.
- Burn Rate returns to normal.
- Error Budget consumption stabilizes.
- Failed requests decrease.
- Monitor status changes from Alert to OK.

---

## Step 9 – Close the Incident

Close the incident only after:

- Service availability is restored.
- Burn Rate is stable.
- Monitors have recovered.
- Service health has been verified.

---

# Verification Checklist

Before closing the incident verify:

- Availability SLO healthy
- Burn Rate normal
- Error Budget stable
- Request success rate restored
- No active critical alerts
- Reliability Dashboard shows healthy status

---

# Escalation Criteria

Escalate the incident if:

- Burn Rate continues increasing after mitigation.
- Availability SLO continues decreasing.
- Error Budget approaches exhaustion.
- Multiple endpoints fail simultaneously.
- Root cause cannot be identified within the response window.

---

# Recovery Indicators

The incident is considered resolved when:

- Availability SLO returns to acceptable levels.
- Burn Rate stabilizes.
- Error Budget consumption returns to normal.
- Traffic success rate recovers.
- All monitors return to **OK** state.

# Lessons Learned

After every P1 incident:

- Perform Root Cause Analysis (RCA).
- Document contributing factors.
- Identify preventive actions.
- Update monitoring if gaps are discovered.
- Improve the runbook based on operational experience.
